# Screen Spec Generator

PRD 파일을 읽고 사용자 흐름에서 화면 목록을 도출한 뒤, 각 화면의 구조 스펙을 `docs/SCREEN_SPEC.md`로 출력한다.

이 스킬은 3단계 파이프라인의 일부를 담당한다:
- `design-system` → 디자인 토큰 정의
- **`screen-spec`** → 화면 구조 정의 (이 스킬)
- `wireframe` → .pen 파일 생성

## 이 스킬이 하는 것

- PRD 분석 → 화면 목록 도출
- 각 화면의 레이아웃 구조, 구성 요소, 표시 데이터, 상태별 화면, 네비게이션 정의
- `docs/SCREEN_SPEC.md` 파일 저장

## 이 스킬이 하지 않는 것

- 디자인 시스템 정의 (색상, 타이포, 간격 토큰 등) → `design-system` 스킬 담당
- 구체적인 px 값, 색상 코드 명시 → 디자인 토큰으로 처리
- .pen 와이어프레임 생성 → `wireframe` 스킬 담당
- 코드 생성

## 작업 흐름

### 1단계: PRD 분석 및 화면 도출

`docs/` 디렉토리에서 PRD 파일(`PRD.md` 또는 `PRD-*.md`)을 찾아 읽는다. `docs/`에 없으면 현재 작업 디렉토리에서도 찾는다.

PRD에서 다음 정보를 추출한다:

- **플랫폼(1.2)**: 웹앱/모바일앱에 따라 레이아웃 패턴 결정
  - 웹앱: NavBar, 다열 그리드, 2-column 레이아웃
  - 모바일앱: 하단 탭바, 단일 컬럼, 풀 너비
- **데이터 모델(1.4)**: 화면에 표시할 데이터의 근거
- **핵심 기능(2.1)**: 기능별 필요 화면
- **예외 처리(2.2 또는 흐름 내 예외)**: 빈 상태/에러 메시지
- **사용자 흐름(2.2 또는 2.3)**: 화면 목록 도출의 핵심 입력

#### 화면 도출 방법

1. 사용자 흐름의 각 단계를 순회한다
2. "시스템이 ~을 보여준다" / "~화면이 표시된다" / "~목록을 보여준다" = 화면
3. 같은 페이지에서 일어나는 연속 단계는 하나의 화면으로 그룹핑
4. 각 화면에 대해: 이름, 목적, 대응 흐름 단계를 정리

#### 공통 컴포넌트 식별

- 웹앱: NavBar (상단 네비게이션)
- 모바일앱: 하단 탭바
- 여러 화면에서 반복되는 요소 (카드, 헤더 등)

#### 사용자 확인

화면 목록과 화면 간 흐름을 사용자에게 보여주고 확인을 받는다.
수정 요청이 있으면 반영한 뒤 다시 확인한다.

### 2단계: 화면별 스펙 작성

사용자가 화면 목록을 승인하면, 각 화면의 상세 스펙을 작성한다.

#### 공통 컴포넌트 먼저 정의

여러 화면에서 반복되는 컴포넌트를 먼저 한 번 정의한다. 각 화면에서는 "공통 NavBar 포함" 등으로 참조한다.

#### 각 화면에 대해 다음을 작성

**목적**: 어떤 흐름의 어떤 단계를 담당하는지

**표시 데이터**: PRD 데이터 모델의 어떤 엔티티/속성을 보여주는지
- 반드시 PRD 1.4 데이터 모델에서 참조
- 예: "상품 엔티티의 이름, 가격, 소분단위, 이미지"

**레이아웃**: 영역 배치 (구조적 표현, 구체적 px 값은 쓰지 않는다)
- 웹앱: 헤더, 콘텐츠(1열/2열 비율), 사이드바, 하단 고정 등
- 모바일앱: 상단바, 콘텐츠, 하단 고정, 하단 탭바 등
- 비율 표현: "좌측 65% / 우측 35%", "2-column", "단일 컬럼" 등

**구성 요소**: 테이블 형식으로 정의

| 요소 | 위치 | 타입 | 동작 |
|------|------|------|------|

- 타입은 의미적으로 표현: "제목 텍스트", "본문 텍스트", "캡션 텍스트", "프로그레스 바", "아이콘 버튼", "주요 버튼" 등
- 크기/색상은 디자인 토큰에서 결정하므로 여기서 명시하지 않는다

**상태별 화면**: PRD의 예외 처리 메시지를 그대로 사용
- 기본 상태: 데이터가 정상 로드된 상태
- 빈 상태: 데이터가 없을 때 (PRD 예외 처리 메시지 사용)
- 로딩 상태: 스켈레톤 UI 또는 로딩 인디케이터
- 에러 상태: 에러 메시지 + 재시도 (PRD 예외 처리 메시지 사용)

**네비게이션**: 이 화면에서 이동 가능한 화면 목록

### 3단계: SCREEN_SPEC.md 저장

`docs/SCREEN_SPEC.md`로 저장한다. `docs/` 디렉토리가 없으면 생성한다.
저장 후 사용자에게 결과를 보여주고 수정사항이 있는지 확인한다.

## SCREEN_SPEC.md 출력 포맷

```markdown
# 화면 스펙: {서비스명}

## 메타정보
- 플랫폼: 웹앱 / 모바일앱

## 화면 목록
| # | 화면명 | 목적 | 대응 흐름 |
|---|--------|------|-----------|

## 화면 흐름도
(텍스트 기반 다이어그램)

## 공통 컴포넌트
### NavBar (웹) / 하단 탭바 (모바일)
(구조, 요소, 동작)

## 화면별 스펙

### 1. {화면명}
#### 목적
#### 표시 데이터
#### 레이아웃
#### 구성 요소
| 요소 | 위치 | 타입 | 동작 |
|------|------|------|------|
#### 상태별 화면
#### 네비게이션
```

## 작성 원칙

- **PRD 충실성**: 사용자 흐름에 있는 화면만 도출한다. 임의로 화면을 추가하지 않는다.
- **데이터 모델 기반**: 표시 데이터는 반드시 PRD 데이터 모델(1.4)에서 참조한다. 데이터 모델에 없는 속성을 표시하겠다고 쓰지 않는다.
- **플랫폼 적합성**: 웹앱이면 웹 패턴(NavBar, 다열 그리드, 2-column), 모바일이면 모바일 패턴(하단 탭바, 단일 컬럼)을 사용한다.
- **구조에 집중**: 구체적인 px 값, 색상 코드, 폰트 크기는 명시하지 않는다. 이는 디자인 토큰에서 결정된다. 레이아웃은 비율과 구조(열 수, 영역 배치)로 표현한다.
- **자족성**: SCREEN_SPEC.md만 읽으면 화면에 무엇이 있고 어떻게 배치되는지 알 수 있어야 한다.
  - **상태별 화면 완전성**: 모든 화면에 기본/로딩/에러 3가지 상태를 반드시 정의한다 (해당 없는 빈 상태 제외). 데이터를 서버에서 로드하는 화면은 로딩/에러 상태가 반드시 있어야 한다.
  - **네비게이션 완전성**: NavBar가 있는 화면은 NavBar 내 모든 링크(로고→홈 등)를 네비게이션에 포함한다. 뒤로가기가 가능한 화면은 뒤로가기 목적지를 명시한다.
  - **PRD 출처 표기 금지**: "(PRD 2.2)", "(PRD 흐름1-4 분기)" 같은 출처 참조를 SCREEN_SPEC.md에 쓰지 않는다. 메시지 내용 자체를 직접 기술하면 되므로 출처 표기는 불필요하다.
- **예외 처리 반영**: PRD의 예외 처리 메시지(에러, 빈 상태 등)를 각 화면의 상태별 화면에 그대로 사용한다. 임의로 메시지를 만들지 않는다.
