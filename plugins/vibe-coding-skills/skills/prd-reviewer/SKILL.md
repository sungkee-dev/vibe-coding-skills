# PRD Reviewer

PRD 파일을 개발자 관점에서 리뷰하여, 이대로 개발에 착수해도 되는지 점검한다.
부족한 부분을 발견하면 사용자에게 어떻게 수정할지 질문하고, 답변을 반영하여 PRD 파일을 직접 수정한다.

## 작업 흐름

### 1단계: PRD 파일 읽기

현재 작업 디렉토리에서 PRD 파일(PRD.md 또는 PRD-*.md)을 찾아 읽는다. 여러 개가 있으면 사용자에게 어떤 파일을 리뷰할지 확인한다.

### 2단계: 체크리스트 기반 리뷰

아래 체크리스트를 기준으로 PRD를 점검한다. 각 항목을 통과(pass) 또는 미흡(needs work)으로 판정한다.

#### 체크리스트

**A. 기능 명세의 구체성**
- 각 기능의 상세 설명이 개발자가 읽고 바로 구현할 수 있을 만큼 구체적인가?
- 사용자 시나리오가 단계별로 명확하게 기술되어 있는가?
- 입력/출력, 상태 변화, 예외 상황이 빠져 있지 않은가?
- 우선순위(P0/P1/P2)가 모든 기능에 부여되어 있는가?

**B. 사용자 흐름의 완성도**
- 핵심 사용자 목표별 흐름이 빠짐없이 정의되어 있는가?
- 각 흐름이 시작부터 완료까지 끊기지 않고 연결되는가?
- 분기점(조건에 따라 달라지는 경로)이 명시되어 있는가?
- 예외 상황(실패, 취소, 되돌아가기)에 대한 처리가 포함되어 있는가?
- 흐름 간 연결(예: 주문 흐름에서 결제 흐름으로의 전환)이 자연스러운가?

**C. 데이터 모델의 충분성**
- 핵심 엔티티가 빠짐없이 정의되어 있는가?
- 각 엔티티의 주요 속성이 구체적으로 나열되어 있는가?
- 엔티티 간 관계(1:N, N:M 등)가 파악 가능한가?
- 사용자 흐름에서 언급되는 데이터가 모두 데이터 모델에 정의되어 있는가?

**D. 엣지 케이스 및 예외 처리**
- 에러 상황(네트워크 실패, 입력 오류 등)에 대한 처리가 기술되어 있는가?
- 빈 상태(데이터 없음, 검색 결과 없음 등)에 대한 안내가 정의되어 있는가?
- 경계값(최소/최대 수량, 글자 수 제한 등)이 명시되어 있는가?

**E. 비기능 요구사항**
- 성능 기준이 구체적인 수치로 제시되어 있는가?
- 보안 요구사항이 기술되어 있는가?
- [TBD]로 남아 있는 항목이 있는가?

**F. 일관성 및 용어**
- 같은 개념에 대해 문서 전체에서 동일한 용어를 사용하고 있는가?
- 기능 간 모순되는 내용이 없는가?
- 데이터 모델의 용어와 사용자 흐름의 용어가 일치하는가?

### 3단계: 리뷰 결과 보고

리뷰 결과를 아래 형식으로 사용자에게 보여준다:

```
## PRD 리뷰 결과

### Pass
- (통과한 항목들을 간략히)

### Needs Work
1. [항목] 구체적으로 무엇이 부족한지 설명
2. [항목] 구체적으로 무엇이 부족한지 설명
...
```

지적 사항이 많을 때는 중요도 순으로 정렬하여 가장 큰 문제부터 보여준다.

### 4단계: 대화형 수정

Needs Work 항목에 대해 사용자에게 하나씩 질문한다. 한 번에 모든 항목을 쏟아내지 않고, 2~3개씩 묶어서 질문한다.

사용자의 답변을 받으면 PRD 파일을 직접 수정한다. 수정 후 변경된 부분을 사용자에게 보여준다.

## 리뷰 원칙

- **개발자의 눈으로 읽는다**: "이 문서만 보고 구현할 수 있는가?"가 핵심 기준이다.
- **모호한 표현을 잡아낸다**: "적절한", "다양한", "사용자 친화적" 같은 표현은 구체적인 기준으로 바꾸도록 제안한다.
- **빠진 것을 찾는다**: 적혀 있는 내용의 문제만이 아니라, 아예 언급되지 않은 필수 요소를 찾아낸다.
- **흐름의 연결성을 검증한다**: 사용자 흐름이 중간에 끊기거나 모호해지는 지점이 없는지 집중적으로 확인한다. 이 정보가 다음 단계(화면 설계)의 핵심 입력이 되기 때문이다.
- **데이터와 흐름의 정합성을 확인한다**: 사용자 흐름에서 "평점을 본다", "배달시간을 확인한다" 같은 언급이 있으면 데이터 모델에 해당 속성이 정의되어 있는지 교차 확인한다.
- **과도하게 지적하지 않는다**: 사소한 문체나 포맷 문제가 아니라, 개발 착수를 가로막는 실질적인 문제에 집중한다.
