# PRD Generator

웹/모바일 애플리케이션을 위한 PRD(Product Requirements Document, 제품 요구사항 문서)를 한국어로 작성하여 Markdown 파일로 저장한다.

PRD는 "무엇을 왜 만드는가"에 집중한다. 화면 설계나 디자인 스펙은 PRD의 범위가 아니다 — 그건 별도의 화면 설계 단계에서 다룬다. 대신, 화면 설계 단계가 참고할 수 있도록 사용자 흐름과 데이터 모델을 충분히 구체적으로 기술한다.

## 작업 흐름

### 1단계: 제품 정보 수집

사용자가 이미 제품에 대해 충분히 설명했다면 바로 PRD 작성으로 넘어간다. 그렇지 않다면 아래 핵심 정보를 파악한다:

- **제품명**: 프로젝트나 기능의 이름
- **플랫폼**: 웹앱 / 모바일앱 / 둘 다 — 이후 화면 설계와 직결되므로 반드시 확인한다
- **배경/문제 정의**: 왜 이 제품이 필요한지, 어떤 문제를 해결하는지
- **대상 사용자**: 누가 쓸 것인지
- **핵심 기능**: 반드시 포함되어야 하는 주요 기능 (3~5개)

한 번에 너무 많은 질문을 하지 않는다. 대화의 맥락에서 이미 파악된 정보는 다시 묻지 않는다.

### 2단계: PRD 작성

아래 템플릿 구조를 따라 PRD를 작성한다. 각 섹션의 내용은 사용자가 제공한 정보를 바탕으로 구체적이고 실행 가능하게 작성한다. 빈말이나 뻔한 문구는 피하고, 개발팀이 읽었을 때 바로 작업에 착수할 수 있을 정도로 명확하게 쓴다.

### 3단계: 파일 저장

작성한 PRD를 현재 작업 디렉토리에 `PRD.md` 형식으로 저장한다.

## PRD 템플릿

```markdown
# {제품명} PRD

## 1. 개요

### 1.1 배경 및 문제 정의
- 현재 상황과 해결하려는 문제를 구체적으로 기술

### 1.2 플랫폼
- 웹앱 / 모바일앱(iOS, Android) / 반응형 웹 등 대상 플랫폼을 명시한다
- 플랫폼에 따라 사용자 흐름의 용어도 달라진다 (예: 웹은 "클릭", 모바일은 "탭")

### 1.3 대상 사용자
- 주요 사용자 그룹과 각 그룹의 특성
- 각 사용자 그룹의 핵심 니즈

### 1.4 핵심 데이터 모델
시스템에서 다루는 주요 엔티티와 속성을 정의한다.
화면 설계 단계에서 "화면에 무엇을 보여줄지" 결정하는 근거가 되므로 빠짐없이 기술한다.

각 엔티티는 다음을 포함:
- 엔티티 이름과 주요 속성
- 엔티티 간 관계 (1:1, 1:N, N:M)

예시:
- **음식점**: 이름, 카테고리, 평점, 배달시간, 최소주문금액, 대표이미지, 영업시간
- **메뉴**: 이름, 가격, 설명, 이미지, 옵션그룹(사이즈/토핑 등)
- **주문**: 주문번호, 상태, 메뉴목록, 총금액, 배달주소, 예상도착시간

#### 엔티티 관계
- 음식점 1 : N 메뉴
- 사용자 1 : N 주문
- 주문 1 : N 주문상품

## 2. 기능 요구사항

### 2.1 핵심 기능
각 기능마다 다음을 포함:
- **기능명**: 한 줄 설명
- **상세 설명**: 기능이 어떻게 동작하는지
- **사용자 시나리오**: 사용자가 이 기능을 어떤 흐름으로 사용하는지
- **우선순위**: P0 (필수) / P1 (중요) / P2 (있으면 좋음)

### 2.2 사용자 흐름
사용자가 핵심 목표를 달성하기 위해 거치는 단계별 흐름을 기술한다.
각 흐름은 시작점부터 완료까지의 전체 경로를 보여준다.

작성 방법:
- 흐름의 이름과 목표를 먼저 명시한다
- 각 단계에서 사용자가 하는 행동과 시스템의 응답을 짝으로 기술한다
- 분기점(조건에 따라 달라지는 경로)을 명시한다
- 예외 상황(실패, 취소, 되돌아가기)도 포함한다

예시:
#### 흐름 1: 음식 주문
**목표**: 사용자가 원하는 음식을 찾아 주문을 완료한다

1. 사용자가 배달 주소를 입력한다 → 시스템이 배달 가능한 음식점 목록을 보여준다
2. 사용자가 카테고리/검색으로 음식점을 탐색한다 → 시스템이 필터링된 결과를 보여준다
3. 사용자가 음식점을 선택한다 → 시스템이 메뉴 목록을 보여준다
4. 사용자가 메뉴를 선택하고 옵션(수량, 사이즈 등)을 지정한다 → 장바구니에 추가된다
   - 분기: 더 주문하려면 3번으로 돌아간다
5. 사용자가 장바구니를 확인하고 결제를 진행한다 → 결제 화면으로 이동
6. 사용자가 결제 수단을 선택하고 결제한다 → 주문이 접수된다
   - 예외: 결제 실패 시 → 다른 결제 수단을 안내한다
7. 시스템이 주문 확인 정보와 예상 배달 시간을 보여준다

## 3. 비기능 요구사항

### 3.1 성능
- 응답 시간, 동시 접속자 수 등 성능 기준

### 3.2 보안
- 인증/인가, 데이터 보호 등 보안 요구사항

### 3.3 확장성
- 향후 확장 가능성 고려 사항

## 4. 기술 스펙 (선택)

### 4.1 기술 스택
- 프론트엔드/백엔드/DB 등 기술 선택과 근거

### 4.2 시스템 아키텍처
- 전체 시스템 구조 개요

### 4.3 API 설계
- 주요 API 엔드포인트와 데이터 모델

## 5. 마일스톤 및 일정

| 단계 | 내용 | 예상 기간 |
|------|------|-----------|
| Phase 1 | MVP 개발 | - |
| Phase 2 | 개선 및 확장 | - |

## 6. 부록

- 용어 정의, 참고 자료, 관련 문서 링크 등
```

## 작성 원칙

- **구체적으로 쓴다**: "사용자 친화적인 UI" 같은 모호한 표현 대신 "회원가입은 3단계 이내로 완료할 수 있어야 한다"처럼 구체적으로.
- **개발자 관점을 고려한다**: 기능 설명은 개발팀이 읽고 바로 구현할 수 있을 만큼 명확하게.
- **우선순위를 명시한다**: 모든 기능에 P0/P1/P2 우선순위를 부여하여 MVP 범위를 명확히 한다.
- **사용자 흐름을 끝까지 연결한다**: 흐름이 중간에 끊기지 않도록 시작부터 완료(또는 실패 처리)까지 전체 경로를 기술한다. 분기점과 예외 상황을 빠뜨리지 않는다.
- **데이터 모델을 빠짐없이 정의한다**: 각 엔티티의 주요 속성을 나열하고, 엔티티 간 관계(1:1, 1:N, N:M)를 명시한다. 이 정보가 이후 화면 설계에서 "무엇을 보여줄지" 결정하는 근거가 된다.
- **데이터와 흐름의 정합성을 확인한다**: 사용자 흐름에서 언급되는 모든 데이터(예: "평점을 본다", "배송 상태를 확인한다")가 데이터 모델에 속성으로 정의되어 있는지 교차 확인한다. 흐름에는 있는데 모델에 없는 데이터가 있으면 모델에 추가한다.
- **없는 정보는 만들어내지 않는다**: 사용자가 제공하지 않은 구체적 수치(일정, 성능 목표 등)는 "[TBD]"로 표기하거나 해당 섹션을 간략히 작성한다.
- **섹션 4(기술 스펙)는 선택**: 사용자가 기술적 내용을 요청하거나 이미 기술 스택에 대해 언급한 경우에만 포함한다. 그렇지 않으면 생략하고, 이후 섹션 번호를 순서대로 매긴다.
- **화면 설계는 하지 않는다**: 화면 레이아웃, 컴포넌트 배치, UI 디자인에 대한 내용은 작성하지 않는다. 그건 다음 단계의 역할이다.
